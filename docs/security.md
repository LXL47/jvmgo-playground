# 在线代码执行安全模型

## 保护目标

平台需要保护 API 可用性、宿主机文件和网络、其他用户任务以及服务资源。用户提交的 Java 源码一律按敌对输入处理，源码内容过滤不属于安全边界。

## 分层边界

1. API 限制请求体和每分钟请求数，不信任客户端提供的转发 IP。
2. Runner 使用固定命令参数调用 `javac`，禁用注解处理器，并校验 class 文件数量和总体积。
3. 每个任务使用独立临时目录，JVM 始终运行在子进程中。
4. JVM 对指令、受管分配、数组和输出计数；Runner 同时实施墙钟超时和输出上限。
5. Runner 容器不映射端口、不挂载宿主机目录，运行在只读文件系统和内部网络中。
6. 生产部署推荐为 Runner 启用 gVisor，降低容器逃逸风险。

## 已知限制

- 受管堆预算统计累计分配，不是垃圾回收后的实时存活对象大小，因此比真实堆上限更严格。
- Java 8 初始化输出流依赖 `sun.misc.Unsafe`。生产 JVM 保留项目原有的纯 Go 模拟内存，但分配会计入预算，模拟地址不能访问宿主机内存。
- Compose 的 CPU、内存和 PID 限制作用于整个 Runner 容器，不是单个请求；并发上限用于避免多个任务共同耗尽容器预算。
- 普通 Docker 不是严格的敌对代码沙箱，不能替代 gVisor、Kata Containers 或虚拟机级隔离。

## 预算调优

默认值先覆盖 HelloWorld、循环、数组排序和基础集合示例，再保留余量。调整时同时观察指令数、执行时间、累计分配和输出，采用正常样例峰值的 3 到 5 倍，不应单独无限提高某一项。

触发输出预算时立即取消进程；触发 JVM 内部预算时返回 `sandbox_limit`；超过墙钟时由 Runner 终止整个进程组并返回 `timeout`。
